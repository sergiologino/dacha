## 2026-03-15 — Yandex OAuth: защита от повторного запуска и диагностика callback

- **Проблема**: вход через Яндекс возвращал пользователя обратно на `/auth/signin` с общей ошибкой `Configuration`, а в серверных логах причина проявлялась как `CallbackRouteError` / `invalid_grant` / `Code has expired`.
- **Реализация**:
  - В `app/auth/signin/signin-form.tsx` добавлены состояния загрузки для провайдеров и блокировка повторного клика по кнопкам входа, чтобы не запускать `signIn("yandex")` несколько раз подряд. Для Яндекса и Google теперь показывается явный текст перехода к провайдеру.
  - На странице входа добавлено дружелюбное сообщение об ошибке: если после попытки `Yandex`-входа возвращается `error=Configuration`, пользователю показывается пояснение, что вход через Яндекс не завершился и можно попробовать ещё раз одним нажатием или временно использовать Google.
  - В `app/api/auth/[...nextauth]/route.ts` обёрнуты auth handlers и добавлен безопасный диагностический лог именно для callback Яндекса: host, forwarded headers, наличие `code/state`, search keys, статус ответа и `location` redirect. Сам код авторизации в логи не пишется.
  - В `auth.ts` добавлен явный `logger` для Auth.js, чтобы ошибки и предупреждения по провайдерам было легче сопоставлять с callback-логами на продакшене.
- **Фикс деплоя**:
  - Для совместимости с текущей версией `Auth.js / NextAuth` в `app/api/auth/[...nextauth]/route.ts` вызовы `handlers.GET` и `handlers.POST` переведены на сигнатуру с одним аргументом (`request`). Ранее передавался лишний `context`, из-за чего падала TypeScript-сборка на деплое.
- **Результат**: UI больше не провоцирует повторный старт авторизации по двойному нажатию, а если проблема останется, по продакшен-логам теперь можно быстро отличить двойной callback, лишний redirect или инфраструктурную проблему.

## 2026-03-15 — Исправление ChunkLoadError после деплоя

- **Проблема**: после деплоя приложение у части пользователей падало с экраном «Что-то пошло не так», а в консоли был `ChunkLoadError` / `Failed to load chunk /_next/static/chunks/... 404`.
- **Причина**: текущий `service worker` кэшировал все `GET`-запросы cache-first, включая HTML и `/_next/static/chunks/*`, при этом использовалось постоянное имя кэша `dacha-ai-v1`. После нового деплоя клиент мог получить старую страницу/старый runtime, который пытался загрузить уже удалённый чанк прошлого билда.
- **Реализация**:
  - В `public/sw.js` изменена стратегия кэширования: `/_next/*` больше не перехватывается и не кэшируется service worker'ом; навигации (`document`) теперь идут network-first с fallback в кэш; кэшируются только безопасные статические ассеты (manifest, icons, images/uploads, styles, fonts). Имя кэша обновлено до `dacha-ai-v2`.
  - В `components/sw-register.tsx` регистрация SW переведена на `updateViaCache: "none"` и сразу запрашивает `registration.update()`, чтобы браузер быстрее подтягивал свежий `sw.js` после релиза.
  - В `app/error.tsx` добавлено специальное восстановление для `ChunkLoadError`: пользователь получает понятный текст про обновление приложения, а кнопка очищает `dacha-ai-*` caches, запрашивает обновление service worker и делает `window.location.reload()`.
- **Результат**: после релиза приложение больше не должно «залипать» на старых чанках между деплоями, а если конфликт кэша всё же возникнет у открытой вкладки, экран ошибки сможет восстановить пользователя в один клик.

## 2026-03-03 — Paywall: кнопка покупки и диагностика ошибок оплаты

- **Проблема**: при нажатии «Купить» в paywall (модальное окно подписки) на десктопе и сервере не происходило заметной реакции; платеж не создавался или падал с ошибкой, при этом пользователь не видел понятного сообщения.
- **Причина**: ошибки на `/api/payments` (невалидное тело запроса, проблемы с ответом YooKassa) обрабатывались обезличенно; на фронтенде кнопка не показывала состояние загрузки, а текст ошибки из бэкенда не отображался явно, поэтому при сбое визуально казалось, что ничего не происходит.
- **Реализация**:
  - В `components/subscribe-modal.tsx` добавлены состояния `isSubmitting` и `error`: кнопка «Оформить Премиум» блокируется и показывает «Создаём платёж...» во время запроса, при ошибках выводится понятное сообщение под кнопкой; текст ошибки берётся из `data.error` ответа `/api/payments`, при отсутствии — показан дружелюбный fallback.
  - В `/api/payments` (`app/api/payments/route.ts`) добавлена валидация тела запроса: безопасный парсинг `request.json()`, проверка `amount` как положительного числа, нормализация `plan` и `description`; при невалидном теле возвращается 400 с явным `error`. Ответ YooKassa теперь проверяется по `response.ok`; при ошибке логируется подробность (`status`, тело), а клиенту возвращается 500 с `error: "Payment creation failed"`.
- **Результат**: при клике по кнопке paywall пользователь видит, что идёт создание платежа; при любых сбоях в конфигурации или ответе YooKassa отображается пояснение, а не «молчаливое» отсутствие реакции, что упрощает отладку и поддерживает доверие к оплате.

## 2026-03-03 — Фото-анализ на мобильном: сжатие изображения перед отправкой

- **Проблема**: с мобильного при отправке фото больного растения на анализ приходила ошибка; с десктопа то же фото обрабатывалось корректно.
- **Причина**: на мобильном камера даёт крупные файлы (3–12 Мп). Фото отправлялось в API как base64 без сжатия. В App Router лимит тела запроса для Route Handlers по умолчанию ~1 MB; `bodySizeLimit` в next.config относится только к Server Actions. При превышении лимита тело обрезалось → на сервере «No image provided» или «Invalid request body» → пользователь видел общую ошибку.
- **Реализация**:
  - Добавлена утилита `lib/compress-image.ts`: сжатие изображения до длинной стороны не более 1200 px и целевого размера ~900 KB (JPEG quality понижается при необходимости), чтобы уложиться в лимит.
  - На странице `/camera`: при выборе/съёмке фото файлы больше 400 KB сжимаются перед отправкой; файлы ≤400 KB отправляются как есть (десктоп, уже сжатые). При ошибке сжатия — fallback на исходный файл через FileReader.
- **Документация**: запись в CHANGELOG_AI.

## 2026-03-01 — Пуш-напоминания: таймзона и крон

- **Проблема**: ручная работа, запланированная на «сегодня», не вызывала пуш-напоминание ни на десктоп, ни в мобильной версии.
- **Причины**: (1) «Сегодня» и «завтра» в кроне считались через `new Date(toLocaleString("en-US", { timeZone }))` — результат парсится в локальном времени сервера, а не в Москве, поэтому границы дня были неверными. (2) В запросе событий не было условия «scheduledDate >= dayStart», плюс события с диапазоном дат (dateTo) могли не попадать в выборку.
- **Реализация**:
  - В `lib/push-reminders.ts` добавлена функция `getDayBoundsInTimezone(date, timeZone)`: возвращает `dayStart` и `dayEnd` в UTC для календарного дня в указанной таймзоне. Для `Europe/Moscow` используются границы с учётом UTC+3.
  - Крон `GET /api/cron/push-reminders` строит «сегодня» и «завтра» через `getDayBoundsInTimezone(..., PUSH_REMINDERS_TZ ?? TZ ?? "Europe/Moscow")`, передаёт в `getReminderEventsByUserForDate(dayStart, dayEnd)`.
  - `getReminderEventsByUserForDate(dayStart, dayEnd)` принимает два аргумента; в запросе Prisma: `scheduledDate: { lte: dayEnd }` и `OR: [ { dateTo: null, scheduledDate: { gte: dayStart } }, { dateTo: { gte: dayStart } } ]` — так попадают и точечные события, и с диапазоном.
  - В `docs/DEPLOY.md` добавлен раздел **8. Пуш-напоминания**: переменные VAPID, CRON_SECRET, PUSH_REMINDERS_TZ; настройка ежедневного крона; проверка «почему не пришло напоминание».
- **Что проверить на продакшене**: заданы ли `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, настроен ли ежедневный вызов `GET /api/cron/push-reminders?secret=...` (например, в 07:00 МСК), включил ли пользователь уведомления в Настройках.

## 2026-02-27 — Редактирование плановых работ на грядках (CRUD + календарь + лунный)

- **Задача**: пользователь должен иметь возможность добавлять, изменять (дата, состав работ) и удалять запланированные работы на грядках; доступ из страницы грядок (клик по событию на таймлайне или кнопка «Добавить работу») и из календаря (клик по карточке или «+ Добавить работу» с выбором растения); те же работы отображать на лунном календаре по датам с возможностью редактирования. Рекомендуемые общие задачи календаря остаются только для чтения.
- **Реализация**:
  - **API**: `POST /api/plants/[id]/timeline/events` — создание события (title, description, scheduledDate, dateTo, isAction, type); `PATCH` и `DELETE /api/plants/[id]/timeline/events/[eventId]` — обновление и удаление. Проверка владельца растения.
  - **Модальное окно** `components/planned-work-modal.tsx`: режимы «добавить» и «редактировать»; поля: название, описание, дата, дата по (опционально), тип (sprout, transplant, water, feed, harvest и др.), чекбокс «это действие»; кнопки Сохранить, Удалить (в режиме редактирования), Отмена. При добавлении из календаря — опциональный выбор грядки/растения (`bedsForPick`).
  - **Грядки** `/garden`: в `PlantTimelineBar` добавлен колбэк `onEventClick`; при выборе события под шкалой отображается кнопка «Изменить»; кнопка «+ Добавить плановую работу» для каждого растения (при наличии таймлайна — под шкалой, при отсутствии — рядом с «Рассчитать таймлайн ухода»). Открытие модалки с контекстом растения и грядки, после сохранения/удаления — инвалидация `beds`.
  - **Календарь** `/calendar`: карточки запланированных работ кликабельны (подсказка «Нажмите, чтобы изменить»); кнопка «+ Добавить работу» в блоке «Запланированные работы» открывает модал в режиме добавления с выбором грядки/растения. В `PlannedWorkItem` добавлены `plantId`, `bedId`, `type` для вызова API.
  - **Лунный календарь**: компонент получает `beds` и `onEditPlannedWork`. Плановые работы по текущему месяцу/году вычисляются через `getPlannedEventsForMonth(beds, month, year)`; на ячейках дней с работами — зелёная точка-индикатор; в карточке выбранного дня секция «Плановые работы с грядок» со списком работ на эту дату, каждая карточка кликабельна и открывает то же модальное окно редактирования.
  - **Общий модуль** `lib/planned-events.ts`: типы `PlannedWorkItem`, функции `getPlannedEventsForMonth(beds, month, year?)` и `getPlannedEventsForDay(plannedItems, dayOfMonth, month, year)` для фильтрации по дню.
