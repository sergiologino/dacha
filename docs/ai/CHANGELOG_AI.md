## 2026-02-27 — Редактирование плановых работ на грядках (CRUD + календарь + лунный)

- **Задача**: пользователь должен иметь возможность добавлять, изменять (дата, состав работ) и удалять запланированные работы на грядках; доступ из страницы грядок (клик по событию на таймлайне или кнопка «Добавить работу») и из календаря (клик по карточке или «+ Добавить работу» с выбором растения); те же работы отображать на лунном календаре по датам с возможностью редактирования. Рекомендуемые общие задачи календаря остаются только для чтения.
- **Реализация**:
  - **API**: `POST /api/plants/[id]/timeline/events` — создание события (title, description, scheduledDate, dateTo, isAction, type); `PATCH` и `DELETE /api/plants/[id]/timeline/events/[eventId]` — обновление и удаление. Проверка владельца растения.
  - **Модальное окно** `components/planned-work-modal.tsx`: режимы «добавить» и «редактировать»; поля: название, описание, дата, дата по (опционально), тип (sprout, transplant, water, feed, harvest и др.), чекбокс «это действие»; кнопки Сохранить, Удалить (в режиме редактирования), Отмена. При добавлении из календаря — опциональный выбор грядки/растения (`bedsForPick`).
  - **Грядки** `/garden`: в `PlantTimelineBar` добавлен колбэк `onEventClick`; при выборе события под шкалой отображается кнопка «Изменить»; кнопка «+ Добавить плановую работу» для каждого растения (при наличии таймлайна — под шкалой, при отсутствии — рядом с «Рассчитать таймлайн ухода»). Открытие модалки с контекстом растения и грядки, после сохранения/удаления — инвалидация `beds`.
  - **Календарь** `/calendar`: карточки запланированных работ кликабельны (подсказка «Нажмите, чтобы изменить»); кнопка «+ Добавить работу» в блоке «Запланированные работы» открывает модал в режиме добавления с выбором грядки/растения. В `PlannedWorkItem` добавлены `plantId`, `bedId`, `type` для вызова API.
  - **Лунный календарь**: компонент получает `beds` и `onEditPlannedWork`. Плановые работы по текущему месяцу/году вычисляются через `getPlannedEventsForMonth(beds, month, year)`; на ячейках дней с работами — зелёная точка-индикатор; в карточке выбранного дня секция «Плановые работы с грядок» со списком работ на эту дату, каждая карточка кликабельна и открывает то же модальное окно редактирования.
  - **Общий модуль** `lib/planned-events.ts`: типы `PlannedWorkItem`, функции `getPlannedEventsForMonth(beds, month, year?)` и `getPlannedEventsForDay(plannedItems, dayOfMonth, month, year)` для фильтрации по дню.

## 2026-02-27 — Запланированные работы с грядок в календаре

- **Задача**: показывать в календаре все работы, запланированные по таймлайнам растений на грядках (дата, грядка, культура, состав работ), визуально отдельно от общих календарных задач; в перспективе — пуш-уведомления по этим работам.
- **Реализация**:
  - На странице `/calendar` в режиме «Задачи» подгружаются грядки пользователя (`useBeds()`). Из `beds` извлекаются события `PlantTimelineEvent` по выбранному месяцу (пересечение диапазона события с месяцем).

- **Дальше**: пуш-уведомления по этим работам (десктоп и мобильная версия).

## 2026-02-27 — Уточнение промптов таймлайна: пересадка только для рассады дома

- **Проблема**: при типе грядки «Теплица» в рекомендованных событиях появлялось «Пересадите рассаду в теплицу», хотя растение уже в теплице.
- **Решение**: обновлены промпты `timeline_system` и `timeline_user` в `prisma/seed.ts` и fallback в `lib/timeline-generate.ts`.
- **timeline_system**: явно указано, что растение уже растёт в указанной грядке; событие «пересадка» добавлять только если тип грядки «рассада дома» (пересадка на постоянное место). Для теплицы, открытого грунта и высокой грядки событий пересадки не включать.
- **timeline_user**: пересадка в списке событий — только для рассады дома; для остальных типов грядки событий пересадки не добавлять.
- После изменений нужно выполнить `npm run db:seed`, чтобы обновить записи в таблице `prompts`.

## 2026-02-27 — Ограничения бесплатного тарифа на грядки, растения и таймлайн

### Контекст и задача
Нужно добавить на главной странице «Мой участок» чёткие ограничения для бесплатных пользователей: не более 2 грядок, 3 растений (всего, по всем грядкам) и таймлайн ухода только для одного растения. При попытке превысить лимит пользователь должен видеть paywall в общем стиле приложения, а в интерфейсе — понятные подписи с количеством доступных действий.

### Backend: лимиты и коды ошибок
- **Beds API** `app/api/beds/route.ts`:
  - В `POST /api/beds` перед созданием грядки добавлена проверка: если `!user.isPremium` и `bed.count({ where: { userId } }) >= 2`, возвращается `402` с телом `{ error: "...не более 2 грядок...", code: "LIMIT_BEDS_FREE" }`.
- **Plants API** `app/api/plants/route.ts`:
  - В `POST /api/plants` добавлена проверка: если `!user.isPremium` и `plant.count({ where: { userId } }) >= 3`, возвращается `402` с телом `{ error: "...не более 3 растений...", code: "LIMIT_PLANTS_FREE" }`.
  - Авто-генерация таймлайна после создания растения теперь зависит от тарифа:
    - Для Премиум-пользователей `generateTimelineForPlant(plant.id)` вызывается всегда.
    - Для бесплатных сначала ищется любое событие `PlantTimelineEvent` у растений этого пользователя; если таймлайна ещё ни у кого нет — вызывается `generateTimelineForPlant`, иначе авто-генерация пропускается.
- **Timeline API** `app/api/plants/[id]/timeline/generate/route.ts`:
  - Для бесплатных пользователей перед вызовом `generateTimelineForPlant` проверяется наличие другого растения с таймлайном: если есть `plantTimelineEvent` c `plant.userId = user.id` и `plant.id != plantId`, маршрут возвращает `402` с `{ error: "таймлайн... только для одного растения", code: "LIMIT_TIMELINE_FREE" }`.

### Frontend: paywall и подсказки в /garden
- **Определение тарифа**:
  - В `GardenContent` добавлены состояния `isPremium` и `showPaywall`; статус тарифа запрашивается через `GET /api/user/premium` при маунте.
  - На основе данных React Query (`usePlants`, `useBeds`) вычисляются: общее число грядок, общее число растений и количество растений с таймлайном (по `timelineEvents.length > 0`).
- **UI и подсказки**:
  - Под заголовком «Мой участок» для бесплатных отображается строка: «Бесплатный тариф: до 2 грядок, 3 растений и 1 таймлайна ухода. Премиум снимает ограничения.»
  - Рядом с кнопкой «Новая грядка» показывается подпись `Грядки: N/2 бесплатно` (для free).
  - В `BedCard` под кнопкой «Добавить растение» отображается подпись `Растения всего: N/3 бесплатно` (для free).
  - Под ссылкой «Рассчитать таймлайн ухода» при отсутствии событий у растения отображается подпись `Таймлайн: N/1 растений бесплатно` (для free, где N — число растений с уже рассчитанным таймлайном).
- **Открытие paywall**:
  - При клике на «Новая грядка» или «Добавить растение» в `BedCard`, если `isPremium === false` и соответствующий лимит уже достигнут (2 грядки или 3 растения), действие не выполняется, вместо этого открывается `<SubscribeModal open={showPaywall} ... />`.
  - При клике на «Рассчитать таймлайн ухода» у растения без событий, если бесплатный пользователь уже имеет таймлайн для другого растения (`timelinePlantsCount >= 1`), генерация не вызывается, а открывается SubscribeModal.

## 2026-02-26 — Анализ фото на таймлайне: вердикт нейросети, синие/красные точки

### Контекст и задача
При фотографировании растения на грядке запускается анализ нейросетью с учётом даты посадки, названия и сорта растения, типа грядки. Нейросеть возвращает вердикт: «всё в норме» (ok) или «есть отклонения» (problem) и текст рекомендаций. Результат отображается на таймлайне точкой (синяя — норма, красная — проблема), по клику выводится текст; вердикт также показывается в галерее под фото.

### Модель и API
- **Photo**: добавлены поля `analysisResult` (текст вердикта), `analysisStatus` (ok | problem), `analyzedAt`. Миграция `20260226200000_add_photo_timeline_analysis`.
- **Промпт** `photo_timeline_verdict` добавлен в таблицу prompts (seed): инструкция для агронома — по фото и контексту (растение, сорт, тип грядки, дата посадки, недель с посадки) вернуть JSON `{ status, verdict }`.
- **lib/analyze-photo-timeline.ts**: `analyzePhotoForTimeline(photoId, imageBase64, mimeType, plant, bed, takenAt, userId)` — вызов AI (vision), парсинг JSON, обновление фото, логирование в ai_call_logs.
- **POST /api/photos**: после сохранения фото вызывается `analyzePhotoForTimeline`; в ответе возвращается фото с заполненными полями анализа (при успехе).

### UI
- **PlantTimelineBar**: новый проп `photoChecks` (массив с id, date, status, verdict). На шкале рисуются синие (ok) и красные (problem) точки по дате съёмки; по клику под шкалой выводится дата и вердикт (аналогично зелёным точкам плановых событий).
- **garden-content**: для каждого растения из `plant.photos` с заполненным `analysisStatus`/`analysisResult` формируется `photoChecks` и передаётся в `PlantTimelineBar`. Таймлайн отображается и при отсутствии плановых событий, если есть хотя бы один фото-вердикт.
- **Галерея фото**: под изображением выводится блок «Вердикт» / «Отклонения» с текстом `analysisResult`; блок виден при просмотре любого фото с проведённым анализом.

### Ссылка «Рассчитать таймлайн ухода» (garden-content.tsx)
- **После нажатия**: кнопка становится некликабельной (`disabled`), текст — «Расчёт…», цвет — нейтральный (серый).
- **При успехе**: после ответа API выполняется refetch грядок; при появлении событий таймлайна ссылка заменяется блоком таймлайна (исчезает).
- **При ошибке**: в `finally` сбрасывается состояние загрузки — кнопка снова кликабельна и с прежним цветом.
- **API**: `onRegenerateTimeline` возвращает `Promise<void>`; вызов обёрнут в `async/await`, ошибки обрабатываются в `catch`/`finally`.
