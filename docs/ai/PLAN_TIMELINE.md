# План: Таймлайн посадок на грядке

## Цель

После добавления растения на грядку рассчитывать **ключевые события роста** и **действия по уходу** (всходы, пересадка, полив, рыхление, освещение/температура для рассады и т.д.). Показывать **графический таймлайн** на блоке растения и **подписи/напоминания** под ним («не забудь взрыхлить рассаду огурцов», «5–7 марта ожидаются первые ростки»). В перспективе — пуши по расписанию.

---

## Исходные данные для расчёта

- **Тип грядки:** открытый грунт / теплица / высокая грядка / **рассада дома** (уже есть в Bed.type: `open`, `greenhouse`, `raised`, `seedling_home`).
- **Растение:** название, привязка к культуре (`cropSlug` или `name`), при наличии — сорт.
- **Дата посадки:** `plant.plantedDate`.
- **Местоположение:** для открытого грунта/теплицы/грядки — регион пользователя (User.region, latitude/longitude) для учёта климата; для **рассада дома** — условия дома (без привязки к региону: освещение, температура в помещении).

---

## Стратегия по нейросети: один запрос при посадке

**Рекомендация: один вызов нейросети сразу после добавления растения** — запросить полный таймлайн событий с датами. Это проще и предсказуемее, чем периодические запросы по расписанию.

**Плюсы:**
- Один запрос на растение, нет крона и очередей.
- Пользователь сразу видит план на весь цикл.
- Даты фиксированы, можно кешировать и показывать офлайн.

**Минусы:**
- Погода/корректировки не подтягиваются автоматически (при желании позже можно добавить «обновить таймлайн» с повторным запросом).

**Альтернатива (по расписанию):** раз в день/неделю дергать нейросеть за «что делать сегодня» по всем растениям — сложнее (очередь, лимиты API, зависимость от доступности сервиса). Имеет смысл только если нужна постоянная подстройка под погоду; для первой версии достаточно одного расчёта при посадке.

## 1. Модель данных

### 1.1. Новая таблица: `PlantTimelineEvent`

Хранит события таймлайна по одному растению.

| Поле           | Тип       | Описание |
|----------------|-----------|----------|
| id             | String    | cuid |
| plantId        | String    | FK → Plant, onDelete Cascade |
| type           | String    | Тип события (см. ниже) |
| title          | String    | Короткий заголовок («Всходы», «Полив», «Пересадка») |
| description    | String?   | Подсказка / текст напоминания |
| scheduledDate  | DateTime  | Одна дата (для точечного события) |
| dateTo         | DateTime? | Конец диапазона (например «5–7 марта») |
| isAction       | Boolean   | true = действие («полить», «взрыхлить»), false = ожидаемое событие («ожидаются всходы») |
| sortOrder      | Int       | Порядок на таймлайне (по дате + порядок) |
| doneAt         | DateTime? | Когда пользователь отметил выполнение (опционально, для действий) |
| createdAt      | DateTime  | |

**Типы событий (type):**  
`sprout` (всходы), `transplant` (пересадка), `water` (полив), `loosen` (рыхление), `light_temp` (освещение/температура, для рассады), `feed` (подкормка), `pinch` (пасынкование и т.п.), `harvest` (урожай), `other`.

Связь: у `Plant` — `timelineEvents PlantTimelineEvent[]`.

### 1.2. Миграция

- Добавить модель `PlantTimelineEvent` в Prisma, связь с `Plant`.
- Создать миграцию `prisma migrate dev` / `migrate deploy`.

## 2. API и нейросеть

### 2.1. Генерация таймлайна при создании растения

**Точка входа:** после успешного `POST /api/plants` (создание растения).

**Варианты реализации:**

- **A) В том же POST /api/plants:** после `prisma.plant.create` вызвать внутреннюю функцию генерации таймлайна (см. ниже). Если нейросеть недоступна или таймаут — растение всё равно создано, таймлайн можно заполнить позже или показать «Таймлайн будет готов через минуту» и дозапросить.
- **B) Отдельный endpoint:** клиент после создания растения вызывает `POST /api/plants/[id]/timeline/generate`. Сервер по plantId достаёт растение + грядку + пользователя, дергает нейросеть, парсит ответ и создаёт записи `PlantTimelineEvent`.

Рекомендация: **A** — сразу при создании, в фоне (не блокировать ответ больше 2–3 сек: запустить генерацию асинхронно и вернуть 201; на клиенте при первом открытии грядки подгружать события, при отсутствии — показать скелетон или «Рассчитываем таймлайн…»).

### 2.2. Запрос к нейросети

**Вход:**  
- Название культуры (и сорт при наличии).  
- Тип грядки: открытый грунт / теплица / высокая грядка / рассада дома.  
- Дата посадки (ISO).  
- Регион или «дома» (для рассады дома не передавать климат региона, указать «рассада в помещении»).

**Промпт (схема):**  
Предложить нейросети вернуть структурированный список событий в формате, удобном для парсинга (JSON), например:

- Для каждого события: тип (всходы, пересадка, полив, рыхление, освещение/температура, подкормка, пасынкование, урожай и т.д.), заголовок, краткое описание, дата или диапазон дат (от–до), флаг «действие» (нужно что-то сделать) или «ожидаемое событие» (информация).

**Пример ожидаемого ответа (упрощённо):**

```json
[
  { "type": "sprout", "title": "Всходы", "description": "Ожидаются первые ростки", "scheduledDate": "2026-03-05", "dateTo": "2026-03-07", "isAction": false },
  { "type": "water", "title": "Полив", "description": "Умеренный полив под корень", "scheduledDate": "2026-03-10", "isAction": true },
  { "type": "loosen", "title": "Рыхление", "description": "Не забудь взрыхлить рассаду огурцов", "scheduledDate": "2026-03-12", "isAction": true },
  { "type": "transplant", "title": "Пересадка", "description": "Высадка в теплицу после закалки", "scheduledDate": "2026-04-20", "isAction": true }
]
```

Парсинг: либо запрашивать у нейросети строго JSON, либо извлекать JSON из markdown/code-block. При ошибке парсинга — логировать, не падать; таймлайн можно оставить пустым или повторить запрос позже.

### 2.3. Эндпоинты

- **GET /api/plants/[id]/timeline** — вернуть список `PlantTimelineEvent` для растения (по plantId и userId). Для отображения таймлайна и подписей под растением.
- **POST /api/plants/[id]/timeline/generate** (опционально) — принудительная перегенерация таймлайна (повторный вызов нейросети). Полезно, если при создании нейросеть была недоступна.
- **PATCH /api/plants/[id]/timeline/[eventId]** (опционально) — отметить действие выполненным (`doneAt = now`).

## 3. UI

### 3.1. Блок растения на грядке

Сейчас блок растения — широкая невысокая строка: имя, дата посадки, фото, кнопки (камера, удалить). Под ней (или внутри блока) нужно добавить:

1. **Графический таймлайн (горизонтальный)**  
   - Ось времени: от даты посадки до последнего события (или до ориентировочной даты урожая/конца ухода).  
   - Метки на оси: ключевые события (всходы, пересадка, полив, рыхление и т.д.) в виде иконок или коротких подписей.  
   - «Линия роста» / текущий день: вертикальная черта или маркер «сегодня», которая сдвигается с каждыми сутками (вычислять на клиенте по `new Date()` относительно диапазона таймлайна).  
   - Компактно: блок широкий и невысокий (например, одна полоска 24–40 px по высоте), события не перекрывают друг друга (разнести по датам).

2. **Подписи под растением**  
   - **Ближайшее действие:** например «Не забудь взрыхлить рассаду огурцов» (из события с `isAction: true` и ближайшей `scheduledDate >= today`).  
   - **Ближайшее ожидаемое:** например «5–7 марта ожидаются первые ростки» (из события с `isAction: false` в ближайшем диапазоне).  
   Можно показывать по одному блоку «Сделать» и «Ожидается», либо один общий «Сейчас: …».

### 3.2. Компоненты

- **PlantTimeline** (или **PlantTimelineBar**): принимает `events: PlantTimelineEvent[]`, `plantedDate: Date`, рисует ось, метки и линию «сегодня». Можно реализовать на CSS (flex/grid) + небольшие абсолютные позиции по датам, либо SVG.
- Под таймлайном — блок текста «Сделать: …» / «Ожидается: …», данные из тех же `events` (отфильтровать по дате и типу).

### 3.3. Загрузка данных

- При открытии грядки вместе с растениями подгружать таймлайны (например, запрос `GET /api/beds` расширить ответ событиями по каждому растению, или отдельный запрос `GET /api/plants/[id]/timeline` для каждого растения — лучше один batch `GET /api/beds?include=timeline` или отдельный `GET /api/plants/timelines?plantIds=...` чтобы не N запросов).
- Если событий нет — не показывать таймлайн или показать «Таймлайн рассчитывается» и кнопку «Обновить» (вызов `POST .../timeline/generate`).

## 4. Напоминания (пуши) и подписи

- **Подписи под растением** — реализуются сразу: вывод текста из `PlantTimelineEvent` (ближайшее действие и ожидаемое событие). Без пушей.
- **Push-уведомления** — отдельный этап: подписка на Web Push, хранение подписок (например в БД), cron или планировщик, который раз в день (или по настройке) смотрит события с `scheduledDate = завтра` / «сегодня» и отправляет пуш «Не забудь взрыхлить рассаду огурцов». В этом плане не расписывается детально; достаточно заложить, что события уже есть в БД с датами и типами, и по ним можно строить рассылку.

## 5. Порядок реализации

1. **Модель и миграция**  
   - Добавить `PlantTimelineEvent`, связь с `Plant`, миграция, `prisma generate`.

2. **Генерация таймлайна (бэкенд)**  
   - Функция «сгенерировать таймлайн»: вход (plantId или plant + bed + user), запрос к AI Integration (чат с промптом и парсинг JSON), сохранение событий в `PlantTimelineEvent`.  
   - Вызов при создании растения (POST /api/plants): после создания растения запускать генерацию (синхронно с таймаутом 5–10 с или асинхронно в фоне).  
   - Опционально: `POST /api/plants/[id]/timeline/generate` для ручного перерасчёта.

3. **API для чтения**  
   - `GET /api/plants/[id]/timeline` — список событий по растению.  
   - Или расширить ответ beds/plants: в составе растения отдавать `timelineEvents` (или отдельный batch-эндпоинт для таймлайнов по списку plantIds).

4. **UI: подписи под растением**  
   - Под блоком растения вывести блок «Сделать» / «Ожидается» на основе ближайших по дате событий из `GET .../timeline`. Без графика — только текст.

5. **UI: графический таймлайн**  
   - Компонент горизонтального таймлайна: ось от посадки до конца, метки событий, линия «сегодня». Подключить к блоку растения.

6. **Опционально: отметка «выполнено»**  
   - PATCH события (doneAt) и скрытие/зачёркивание выполненных действий в UI.

7. **Позже: Push-уведомления**  
   - Отдельная задача: подписка, планировщик, отправка пушей по событиям с датой «сегодня»/«завтра».

## 6. Риски и упрощения

- **Нейросеть возвращает не JSON:** промпт явно просить «только JSON, без markdown»; при неудаче парсинга — не ломать создание растения, таймлайн оставить пустым.  
- **Лимиты AI:** один запрос на одно добавление растения; при массовом добавлении — по одному запросу на растение (очередь не обязательна в первой версии).  
- **Рассада дома:** в промпте явно указать «рассада в помещении, учитывай освещение и температуру дома, без привязки к региону».  
- **Нет привязки к культуре (cropSlug):** если растение добавлено «произвольным именем» без справочника — передать в нейросеть только имя и тип грядки; качество дат может быть ниже, но хотя бы ориентировочный план можно выдать.

## 7. Итог

- **Один запрос к нейросети при посадке** — получить полный таймлайн событий с датами и сохранить в `PlantTimelineEvent`.  
- **Хранение:** таблица событий по растению, типы: всходы, пересадка, полив, рыхление, освещение/температура и т.д.  
- **UI:** широкий невысокий графический таймлайн на блоке растения + подписи «Сделать» / «Ожидается» под ним.  
- **Пуши** — следующий этап, после появления стабильных событий и подписей в UI.
