# Деплой «Любимая Дача» на сервер

Инструкция с учётом миграций БД и переменных окружения.

---

## 1. Подготовка

### 1.1. База данных

- **PostgreSQL** должен быть развёрнут и доступен по сети.
- Строка подключения: `DATABASE_URL=postgresql://USER:PASSWORD@HOST:5432/DATABASE`

#### DBeaver / внешний клиент: «неверный пароль», хотя приложение работает

- В коде **нет подмены** пароля БД: Prisma берёт **`DATABASE_URL` у процесса Node/контейнера приложения как есть**.
- Логин, пароль, хост, порт и имя БД — из разбора этой строки (`USER`, `PASSWORD`, `HOST`, `PORT`, `DATABASE`). Если в URL пароль **URL-encoded** (например `@` записан как `%40`), в DBeaver в поле «Password» нужен **уже декодированный** пароль, если вы вводите поля отдельно.
- Символ **`@` в пароле** ломает формат `postgresql://user:pass@host` без кодирования — тогда в `DATABASE_URL` пароль обязан быть в encoded-виде; при ручном вводе в DBeaver используйте **сырой** пароль.
- Если приложение в **Docker / Coolify**, с ноутбука подключайтесь к **хосту и порту, доступным снаружи** (проброс на хост), а не к внутреннему имени вроде `postgres` из compose.
- Пароль из **`POSTGRES_PASSWORD` контейнера Postgres** и пароль в **`DATABASE_URL` приложения** могут **не совпадать**, если их задавали раздельно. Для DBeaver ориентируйтесь на **`DATABASE_URL` того окружения, где реально крутится приложение** (или заведите отдельного read-only пользователя в Postgres).

### 1.2. Переменные окружения

Соберите значения для **сборки** и **рантайма**.

**Обязательные при сборке (NEXT_PUBLIC_* вшиваются в бандл):**

| Переменная | Описание |
|------------|----------|
| `NEXT_PUBLIC_GA_ID` | ID счётчика Google Analytics (например `G-XXXXXXXX`) |
| `NEXT_PUBLIC_YM_ID` | ID счётчика Яндекс.Метрики |
| `DATABASE_URL` | Строка подключения к PostgreSQL (для `prisma generate` при сборке можно dummy) |

**Обязательные при запуске (рантайм):**

| Переменная | Описание |
|------------|----------|
| `DATABASE_URL` | Реальная строка подключения к PostgreSQL |
| `AUTH_SECRET` | Секрет NextAuth (`openssl rand -base64 32`) |
| `NEXTAUTH_URL` | URL приложения (например `https://dacha-ai.ru`) |
| `AUTH_GOOGLE_ID`, `AUTH_GOOGLE_SECRET` | OAuth Google |
| `AUTH_YANDEX_ID`, `AUTH_YANDEX_SECRET` | OAuth Яндекс |
| `SMS_RU_API_ID` | API ID для отправки одноразовых кодов через SMS.ru |
| `AI_INTEGRATION_URL`, `AI_INTEGRATION_API_KEY` | AI-интегратор |
| `YOOKASSA_SHOP_ID`, `YOOKASSA_SECRET_KEY` | ЮKassa (оплаты) |
| `ADMIN_EMAILS` | Список email админов через запятую (для админки и платежей) |

**Опционально (рантайм):**

| Переменная | Описание |
|------------|----------|
| `PHOTOS_UPLOAD_DIR` | Абсолютный путь к папке с файлами фото на диске (см. **§3.6**). По умолчанию приложение пишет и читает **`/app/public/uploads`** в Docker (`cwd` + `public/uploads`). |
| `PLANT_PHOTOS_INLINE` | `1` / `true` — **не писать** снимки на диск: JPEG сохраняется в БД как `data:image/jpeg;base64,...` (надёжно без Docker-тома; база тяжелее). См. **§3.6**. |
| `NEXT_PUBLIC_FEEDBACK_TELEGRAM_URL` | Ссылка на Telegram для обратной связи (отдаётся через `/api/config`) |
| `NEXT_PUBLIC_FEEDBACK_MAX_URL` | Ссылка на MAX для обратной связи |
| `NEXT_PUBLIC_YANDEX_VERIFICATION` | Код верификации в Вебмастере Яндекса |
| `NEXT_PUBLIC_ENABLE_PHONE_AUTH` | `1` чтобы показать вход по телефону на UI; по умолчанию блок скрыт до готовности провайдера |
| `SMS_RU_FROM` | Подпись отправителя в SMS.ru, если она подключена в аккаунте |
| `SMS_RU_TEST` | `1` для тестового режима SMS.ru без реальной отправки |
| `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY` | Ключи Web Push (для пуш-напоминаний о работах и погодных предупреждений). Сгенерировать: `npx web-push generate-vapid-keys` |
| `CRON_SECRET` | Секрет для вызова cron-задач пушей (GET `/api/cron/push-reminders?secret=...`, `/api/cron/weather-alerts?secret=...` или заголовок `Authorization: Bearer ...`) |
| `PUSH_REMINDERS_TZ` | Часовой пояс для «сегодня/завтра» (по умолчанию `Europe/Moscow`) |

## 2. Миграции БД

Миграции нужно применить **до первого запуска** приложения (или при каждом обновлении, если появились новые миграции).

### Вариант A: на сервере (рекомендуется)

На машине, с которой есть доступ к продакшен-БД:

```bash
# В корне проекта (или там, где лежит prisma/)
export DATABASE_URL="postgresql://USER:PASSWORD@HOST:5432/DATABASE"

# Применить все миграции из prisma/migrations/
npx prisma migrate deploy
```

Успешный вывод: `Applied X migration(s).`

### Coolify: терминал контейнера **приложения** (не Postgres)

Образ собирается как **Next.js standalone**: в рантайме до недавнего времени не было папки `prisma/` и CLI Prisma, поэтому `npx prisma` тянул **Prisma 7** с npm и падал на `prisma.config.ts` / отсутствии `dotenv`. В актуальном `Dockerfile` в образ добавляются `prisma/`, `prisma.config.ts` и пакеты `prisma@6.19.0`, `tsx`, `dotenv` в `/opt/prisma-cli`, **симлинк** `/app/node_modules/prisma` → эта установка (для `prisma/config` в `prisma.config.ts`) и **`NODE_PATH=/opt/prisma-cli/node_modules`** (чтобы `tsx prisma/seed-guide-content.ts` находил `dotenv/config` и прочие пакеты из той же установки).

После **пересборки и деплоя** образа выполните в терминале **сервиса приложения**:

```bash
cd /app
npx prisma migrate deploy
npm run db:seed:guide
```

Используйте локальный CLI (`npx prisma` подхватит `node_modules` образа), не соглашайтесь на установку «новой» версии Prisma в интерактиве npx, если терминал снова спросит (после фикса Dockerfile этого быть не должно).

### Вариант B: через Docker (одноразовый контейнер)

Если на сервере нет Node, можно запустить миграции из временного контейнера:

```bash
docker run --rm \
  -e DATABASE_URL="postgresql://USER:PASSWORD@HOST:5432/DATABASE" \
  -v "$(pwd)/prisma:/app/prisma" \
  -w /app \
  node:22-alpine \
  sh -c "npm init -y && npx prisma migrate deploy"
```

Либо использовать образ приложения и выполнить команду внутри него (если в образе есть `prisma` и `migrations`).

### Если миграций ещё не было (первый деплой)

При первом деплое на пустую БД:

```bash
export DATABASE_URL="postgresql://..."
npx prisma migrate deploy
```

Это создаст все таблицы по текущим миграциям в `prisma/migrations/`.

### Если что-то пошло не так

- **Ошибка «migration failed»:** проверьте логи, откатите при необходимости вручную или через `prisma migrate resolve`.
- **Нужно только «подтянуть» схему без миграций** (для быстрого прототипа):  
  `npx prisma db push`  
  В продакшене предпочтительно использовать `migrate deploy`.

## 3. Деплой через Docker

### 3.1. Сборка образа

В корне репозитория:

```bash
docker build \
  --build-arg NEXT_PUBLIC_GA_ID="G-XXXXXXXX" \
  --build-arg NEXT_PUBLIC_YM_ID="XXXXXXXX" \
  --build-arg DATABASE_URL="postgresql://dummy:dummy@localhost:5432/dummy" \
  -t dacha-ai:latest .
```

- `DATABASE_URL` при сборке может быть заглушкой (нужен для `prisma generate`).
- Реальный `DATABASE_URL` задаётся при запуске контейнера (см. ниже).
- Для билда уже задан `NODE_OPTIONS=--max-old-space-size=4096` в Dockerfile (снижает риск падения при генерации 100+ SSG страниц).

### 3.2. Запуск контейнера

```bash
docker run -d \
  --name dacha-ai \
  -p 3000:3000 \
  -e DATABASE_URL="postgresql://USER:PASSWORD@HOST:5432/DATABASE" \
  -e AUTH_SECRET="..." \
  -e NEXTAUTH_URL="https://dacha-ai.ru" \
  -e AUTH_GOOGLE_ID="..." \
  -e AUTH_GOOGLE_SECRET="..." \
  -e AUTH_YANDEX_ID="..." \
  -e AUTH_YANDEX_SECRET="..." \
  -e AI_INTEGRATION_URL="..." \
  -e AI_INTEGRATION_API_KEY="..." \
  -e YOOKASSA_SHOP_ID="..." \
  -e YOOKASSA_SECRET_KEY="..." \
  -e ADMIN_EMAILS="admin@example.com" \
  dacha-ai:latest
```

Остальные переменные (обратная связь, верификация и т.д.) при необходимости добавьте через `-e`.

**При старте контейнера** (актуальный `Dockerfile`) сначала **`psql`** применяет к `_prisma_migrations` идемпотентную починку (NULL/`applied_steps_count`, дубли по `migration_name`) — иначе Prisma падает при чтении истории ещё до выполнения SQL из `prisma/migrations`. Затем **`npx prisma migrate deploy`**, затем **`npx tsx prisma/seed-guide-content.ts`** (лайфхаки и интересные факты, идемпотентный upsert), после чего **`node server.js`**. Отдельно заходить в терминал нужно только если миграция упала и требуется разбор (см. ниже). В **Coolify** не подменяйте команду запуска на `node server.js` вручную — оставьте дефолт образа (`docker-entrypoint.sh`), иначе авто-миграции не выполнятся.

**Если после обновления репозитория Prisma сообщает о несовпадении checksum для уже применённой миграции `20260316090000_add_phone_auth`** (в файле добавлена починка `_prisma_migrations` и идемпотентный DDL): на этой базе однократно выполните `npx prisma migrate resolve` по [документации Prisma](https://www.prisma.io/docs/guides/migrate/production-troubleshooting) или обновите checksum в `_prisma_migrations` — это нужно только окружениям, где эта миграция **уже была успешно** применена со **старым** содержимым SQL.

**Если миграция `20260316090000_add_phone_auth` в статусе failed (P3018)** после прошлых попыток: в терминале приложения один раз `npx prisma migrate resolve --rolled-back "20260316090000_add_phone_auth"`, затем снова деплой / `migrate deploy` — дальше подтянется обновлённый SQL (починка истории + идемпотентные `IF NOT EXISTS`).

**Диагностика через `psql`:** переменная **`DATABASE_URL` есть в контейнере приложения**, в контейнере **PostgreSQL** её часто нет — тогда `psql "$DATABASE_URL"` подключается к сокету и к базе с именем как у пользователя (например `dacha_db_admin`), отсюда `database "dacha_db_admin" does not exist`. В контейнере БД используйте **`psql -U "$POSTGRES_USER" -d "$POSTGRES_DB"`** (значения из настроек сервиса Postgres в Coolify), либо задайте полный URL вручную.

**P3009 (failed migrations):** в `_prisma_migrations` остаётся строка с пустым `finished_at` после падения миграции — Prisma отказывается применять новые. В entrypoint перед `migrate deploy` такие строки удаляются (на старте контейнера параллельной миграции нет); затем миграция с идемпотентным SQL (например `20260316090000_add_phone_auth`) может безопасно выполниться снова.

### 3.6. Фото растений: постоянное хранилище (volume)

Файлы с камеры сохраняются на **диск контейнера**. Если при деплое контейнер каждый раз создаётся **заново** без тома, папка с картинками **обнуляется** — в БД остаётся путь `/uploads/…`, а байтов уже нет → превью и `/api/photos/…/image` дают **404**.

**Самый простой вариант (рекомендуется):** примонтировать **именованный том** или каталог хоста **ровно в путь `/app/public/uploads`** внутри контейнера. Тогда:

- загрузка (`POST /api/photos`) и отдача (`GET /api/photos/…/image`) используют **одну и ту же** папку;
- при необходимости те же файлы отдаёт Next как статика по URL **`/uploads/…`** из `public/uploads`;
- **переменную `PHOTOS_UPLOAD_DIR` можно не задавать**.

Пример запуска с томом Docker:

```bash
docker volume create dacha-ai-uploads

docker run -d \
  --name dacha-ai \
  -p 3000:3000 \
  -v dacha-ai-uploads:/app/public/uploads \
  -e DATABASE_URL="postgresql://..." \
  -e AUTH_SECRET="..." \
  -e NEXTAUTH_URL="https://ваш-домен" \
  # … остальные -e как в §3.2 …
  dacha-ai:latest
```

Если удобнее папка на сервере (VPS):

```bash
mkdir -p /srv/dacha-ai/uploads
docker run -d \
  --name dacha-ai \
  -p 3000:3000 \
  -v /srv/dacha-ai/uploads:/app/public/uploads \
  …
```

**Депой из Git на PaaS (Timeweb Cloud, Render, Railway, Coolify и т.п.):** ищите в панели пункт вроде **Volumes**, **Persistent storage**, **Диск**, **Непрерывный том**. Укажите:

- **mount path внутри контейнера:** `/app/public/uploads`
- **размер** — хотя бы 1–5 ГБ под фото

Секреты и `DATABASE_URL`, как правило, задаются в том же интерфейсе, что и сейчас. После добавления тома сделайте **редеплой** (новый запуск контейнера).

**Если провайдер не умеет тома** — задайте **`PLANT_PHOTOS_INLINE=1`** (или `true`): новые загрузки попадут в PostgreSQL как data URL и будут открываться через `/api/photos/…/image` даже без файлов на диске. Старые записи с путём `/uploads/…`, у которых файл уже потерян, **восстановить нельзя** — только загрузить фото заново или поднять бэкап папки `uploads`.

**Важно:** раньше при отсутствии файла API делал редирект на `/uploads/…` в браузере (ложный 404 в DevTools). Сейчас отдаётся 404 с того же API — включите том **или** `PLANT_PHOTOS_INLINE`.

**Переменная `PHOTOS_UPLOAD_DIR`:** нужна только если том монтируете **не** в `/app/public/uploads`, а, например, в `/data/uploads`. Тогда задайте **и** том с этим путём **и** `-e PHOTOS_UPLOAD_DIR=/data/uploads`. Значение должно **совпадать** с тем, куда реально пишется диск при загрузке.

**Nginx перед приложением:** отдельная настройка `location /uploads/` нужна **только если** nginx сам отдаёт файлы с диска. Если nginx только **проксирует всё** на `http://127.0.0.1:3000` (или на контейнер), достаточно тома **`/app/public/uploads`** — ответы по `/uploads/…` и API отдаёт Next.js.

### 3.3. Обновление (редеплой)

1. Применить миграции (если появились новые):
   ```bash
   export DATABASE_URL="postgresql://..."
   npx prisma migrate deploy
   ```
2. Собрать новый образ (с теми же build-arg, что и выше).
3. Остановить и удалить старый контейнер:  
   `docker stop dacha-ai && docker rm dacha-ai`
4. Запустить новый контейнер с теми же `-e` (или использовать docker-compose).

### 3.4. Ошибка `ERROR: failed to load ref: … not found` при сборке на PaaS

Такое сообщение чаще всего связано **не с ошибкой в Dockerfile**, а с **битым или неполным кэшем слоёв** у удалённого билдера (BuildKit): следующий этап не может подтянуть артефакт предыдущей стадии. Multi-stage сборка (`COPY --from=deps`, `--from=builder`) этому чувствительна.

**Что сделать (по приоритету):**

1. **Timeweb Cloud (и аналоги):** в настройках приложения откройте блок сборки (**Build** / **Build Settings**) и запустите **пересборку с очисткой кэша** (например **Rebuild** + **Clean cache** / «с нуля»), если в интерфейсе есть такая опция.
2. Если опции нет: внесите **любое тривиальное изменение** в `Dockerfile` (например комментарий с датой) и **закоммитьте и запушьте** — часто это запускает новую сборку без переиспользования старого кэша.
3. **Проверка локально** (убедиться, что образ собирается):
   ```bash
   docker build --no-cache \
     --build-arg NEXT_PUBLIC_GA_ID="G-XXXXXXXX" \
     --build-arg NEXT_PUBLIC_YM_ID="XXXXXXXX" \
     --build-arg DATABASE_URL="postgresql://dummy:dummy@localhost:5432/dummy" \
     -t dacha-ai:test .
   ```
   Если локально `--no-cache` проходит, а на платформе — нет, проблема почти наверняка в **кэше/инфраструктуре билда** у хостинга; повторите шаг 1 или обратитесь в поддержку с логом.

### 3.5. Alpine / `apk`: `DNS: transient error` при установке `libc6-compat`

На этапе `apk add --no-cache libc6-compat` билдер может увидеть **временный сбой DNS** или недоступность CDN репозитория Alpine. Часто помогает **просто перезапустить сборку**.

В образе уже стоят **несколько повторов** `apk add` с паузой (см. стадию `deps` в `Dockerfile`). Базовый образ — **`node:22-alpine`** без привязки к конкретному `alpine3.xx`: тянется актуальная линия Node 22 для Alpine, что снижает риск застрять на одной «проблемной» минорной версии дистрибутива.

Если ошибка не исчезает после повторов и редеплоев — имеет смысл **написать в поддержку Timeweb** с фрагментом лога (возможны ограничения сети/DNS у их билд-раннера). Альтернатива архитектурно — перейти на Debian-based образ (`node:22-bookworm-slim`) и убрать зависимость от `apk` на стадии deps; это уже заметное изменение стека.

## 5. Чек-лист перед продакшеном

- [ ] Смонтирован постоянный том (или каталог) в **`/app/public/uploads`**, иначе загруженные фото пропадут после редеплоя (см. **§3.6**)
- [ ] Миграции применены: `npx prisma migrate deploy`
- [ ] `NEXTAUTH_URL` совпадает с реальным доменом (возврат после оплаты и логин)
- [ ] `ADMIN_EMAILS` задан (для админки и вкладки «Платежи» в настройках)
- [ ] В сборку переданы `NEXT_PUBLIC_GA_ID` и `NEXT_PUBLIC_YM_ID` (если нужна аналитика)
- [ ] ЮKassa: на продакшене используются боевые `YOOKASSA_SHOP_ID` и `YOOKASSA_SECRET_KEY`
- [ ] Пуш-напоминания: заданы `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`; настроены cron-вызовы `/api/cron/push-reminders` и `/api/cron/weather-alerts` с `CRON_SECRET` (см. раздел 8)

## 6. Краткая последовательность (рекомендуемая)

1. Развернуть PostgreSQL, получить `DATABASE_URL`.
2. Клонировать репозиторий на сервер (или собирать образ в CI).
3. Выполнить миграции:  
   `DATABASE_URL="..." npx prisma migrate deploy`
4. Собрать образ (Docker) или `npm run build` (без Docker).
5. Запустить приложение с полным набором переменных окружения.
6. Проверить: `https://ваш-домен/api/health` → `{"status":"ok"}`.

После этого можно открывать сайт, логиниться и при необходимости проверить админку (Настройки → Платежи) под учётной записью из `ADMIN_EMAILS`.

## 7. Не вижу вкладку «Платежи» в Настройках

Вкладка **«Платежи»** (и блок «Управление (Админ)») показываются **только если ваш email входит в список админов**.

1. **Где смотреть:** Настройки — иконка профиля (аватар) в шапке справа → «Настройки», или URL `/settings`.
2. **Проверьте на сервере переменную `ADMIN_EMAILS`:**
   - Должна быть задана при запуске контейнера/процесса (рантайм), например:  
     `ADMIN_EMAILS=your@email.com` или `ADMIN_EMAILS=admin1@mail.ru,admin2@gmail.com`.
   - Регистр букв в email не важен (сравнение идёт по нижнему регистру).
   - Пробелы вокруг запятых допустимы — они обрезаются.
3. **После изменения переменной** перезапустите приложение и при необходимости обновите страницу Настроек или перелогиньтесь.
4. Если после этого вкладки «Профиль» и «Платежи» по-прежнему нет — проверьте, что в приложении вы залогинены под тем же email, который указан в `ADMIN_EMAILS`.

## 8. Пуш-напоминания о плановых работах

Напоминания о задачах на «сегодня» и «завтра» отправляются по Web Push, если выполнено:

1. **Ключи VAPID** заданы в окружении: `VAPID_PUBLIC_KEY` и `VAPID_PRIVATE_KEY`. Сгенерировать: `npx web-push generate-vapid-keys`.
2. **Пользователь включил уведомления** в Настройках (кнопка «Включить уведомления»). Без подписки пуши не отправляются.
3. **Крон-задача вызывается каждый день** (например, в 07:00 по Москве).

### 8.0 Откуда взять `CRON_SECRET` и как его использовать

**`CRON_SECRET` не генерируется автоматически** — это произвольная длинная строка, которую вы сами задаёте в окружении приложения и **тем же значением** подставляете в URL или заголовок при вызове крона (чтобы посторонние не дергали ваши эндпоинты).

**Сгенерировать надёжный секрет (любой из вариантов):**

- Linux / macOS / Git Bash на Windows:
  ```bash
  openssl rand -base64 32
  ```
  или:
  ```bash
  openssl rand -hex 32
  ```
- Node.js:
  ```bash
  node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
  ```

**Куда записать:** в переменные окружения сервера (панель хостинга, `docker-compose`, секреты CI), например:

```env
CRON_SECRET="вставьте_сюда_одну_строку_из_openssl_rand_или_node"
```

Перезапустите приложение. В логах секрет не светите.

**Как вызывать крон с этим секретом:**

| Эндпоинт | Зачем | Как часто (рекомендация) |
|----------|--------|---------------------------|
| `GET /api/cron/push-reminders` | Напоминания о работах на сегодня/завтра | Раз в день (например 07:00 МСК) |
| `GET /api/cron/weather-alerts` | Погодные предупреждения (кто включил в профиле) | Каждые 15 минут |

**Варианты авторизации** (должно совпадать с `CRON_SECRET` в env приложения):

1. Query-параметр:  
   `https://ваш-домен/api/cron/push-reminders?secret=ВАШ_CRON_SECRET`
2. Заголовок:  
   `Authorization: Bearer ВАШ_CRON_SECRET`

На **планировщике** (TimeWeb, cron на VPS, GitHub Actions по расписанию, EasyCron и т.д.):

- Создайте задачу с методом **GET**.
- Вставьте полный URL с `?secret=...` **или** настройте заголовок `Authorization: Bearer ...`, если интерфейс это поддерживает.
- **Не кладите секрет в публичный репозиторий.** В cron у многих хостингов есть скрытое поле «заголовки» / «environment» — используйте его.

**Если `CRON_SECRET` в приложении не задан:** эндпоинты крона принимают вызов **без проверки** (удобно для отладки; на продакшене секрет задайте обязательно).

### Настройка крона

Эндпоинт: `GET https://ваш-домен/api/cron/push-reminders`

**Авторизация:** передать секрет одним из способов:
- Query: `?secret=ВАШ_CRON_SECRET`
- Заголовок: `Authorization: Bearer ВАШ_CRON_SECRET`

Переменная `CRON_SECRET` должна быть задана при запуске приложения. Если не задана, эндпоинт принимает любой запрос (удобно для проверки, на продакшене лучше задать секрет).

**Пример (TimeWeb, cron раз в день в 07:00 МСК):**
```bash
0 7 * * * curl -s "https://dacha-ai.ru/api/cron/push-reminders?secret=$CRON_SECRET"
```

Или через планировщик хостинга (cron job) с методом GET и URL с параметром `secret`.

### Почему не пришло напоминание

- **Крон не настроен** — запланируйте ежедневный вызов `GET /api/cron/push-reminders` с секретом.
- **Уведомления не включены** — в Настройках нажать «Включить уведомления» (и разрешить в браузере). Подписка привязана к устройству/браузеру; на каждом устройстве нужно включить отдельно.
- **VAPID не заданы** — без них пуши не отправляются; в ответе крона будет 503 «Push not configured».
- **Дата события** — «сегодня» и «завтра» считаются по часовому поясу `PUSH_REMINDERS_TZ` (по умолчанию Europe/Moscow). Событие должно быть с `isAction: true` и без `doneAt`. В выборку попадают и работы из календаря ухода, и события, добавленные пользователем вручную (`isUserCreated: true`).
- **Повторы крона** — одинаковое уведомление не должно уйти повторно: cron дедуплицируется через таблицу `push_delivery_logs`. Если подписок у пользователя нет, уведомление не фиксируется как отправленное.

### 8.1 Погодные предупреждения

Погодные push-уведомления о заморозках, сильном ветре, дожде, снеге и жаре отправляются отдельным cron-эндпоинтом:

`GET https://ваш-домен/api/cron/weather-alerts`

Условия работы:

1. Заданы `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY` и `CRON_SECRET`.
2. Пользователь включил обычные push-уведомления в профиле.
3. Пользователь сохранил местоположение участка и включил в профиле «Погодные предупреждения».
4. Cron запускается регулярно, лучше раз в 15 минут, чтобы поддерживать минимальный пользовательский интервал проверки.

**Рекомендуемый cron для погоды (раз в 15 минут):**
```bash
*/15 * * * * curl -s "https://dacha-ai.ru/api/cron/weather-alerts?secret=$CRON_SECRET"
```

Важно:

- У каждого пользователя свой интервал проверки в профиле: от `15 минут` до `6 часов`, по умолчанию `1 час`.
- Если погода не изменилась по значимым рискам, новый push не отправляется.
- При смене местоположения состояние прошлых погодных предупреждений сбрасывается, чтобы система пересчитала риски заново.
